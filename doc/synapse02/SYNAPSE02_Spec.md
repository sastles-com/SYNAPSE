## 1. 目的

SYNAPSE02 は、`SYNAPSE_LoopArchitecture.md` が定義する**予測開発ループ**（propose → compose → predict → measure error → update）を、NV-2026 車両開発を題材に**体験・説明できるモック駆動デモポータル**である。

- **対象ユーザー:** 経営層・各ドメインのマネージャ・現場エンジニア（社内プレゼン/合意形成の場で操作しながら説明する用途）
- **成功条件:** 見た人が「ループとは何か」「4層スタックと何が違うのか」「自分のドメインはどこに関与するのか」を10分の操作で理解できること
- **synapse01 との関係:** 置換ではなく並置。synapse01＝4層スタック時代のポータル、synapse02＝ループ時代のポータル。両者を見比べられること自体がデモ価値を持つ

## 2. スコープ / 非スコープ

**スコープ（やる）:**
- 16週分のデモシナリオデータ（モック）を時系列再生するSPA
- ループ5要素すべての可視化（観測 o／解釈 z／合意ダイナミクス／共通図 w／答え合わせ）
- MHNG 提案-受容ラウンドのステップ再生シミュレータ（シナリオデータの再生。計算はしない）
- collapse operator（焼きなまし）・staleness decay・collapse certificate・fossil の可視化
- ループ健全性メトリクス7指標（`SYNAPSE_LoopArchitecture.md` §8）
- 3Dアンカー（synapse01 の Viewer3D を流用し、ループ用の着色モードを追加）

**非スコープ（やらない）:**
- バックエンド・API・永続化（synapse01 同様、`data.js` のエクスポート定数が全データ源）
- 本物の MHNG 計算（尤度計算・LLMエージェント）。受容確率などの数値はすべてシナリオデータに事前記録された値を表示する
- 認証・マルチユーザー・編集機能（閲覧と再生操作のみ）
- テスト・リンタ（プロトタイプ段階、synapse01 と同基準）

## 3. 技術スタック

synapse01 と完全に同一とする（流用効率と保守一貫性のため）:

- React 18 + Vite、Three.js（3Dタブのみ）
- 状態管理ライブラリなし（ルートコンポーネントの useState + props）
- CSSフレームワークなし（**全スタイルはインライン style オブジェクト**）
- デザイントークンは `data.js` の `C` / `STATUS_HEX` / `STATUS_3JS` 方式を踏襲（synapse02 用に拡張。DataModel 参照）

```bash
cd synapse02
npm install
npm run dev      # http://localhost:5174 （synapse01 と同時起動できるようポートをずらす）
npm run build
```

## 4. グローバルUI構造

### 4.1 レイアウト

synapse01 の `SynapsePortal.jsx` と同構造：ヘッダー（ロゴ + タブ列 + シナリオ時刻表示）＋タブ切替の単純 switch。

**タブ構成（6タブ）:**

| # | タブ名（表示） | 英名（内部ID） | 対応概念 |
|---|---|---|---|
| 0 | ループ | `loop` | ループ全景＋健全性メトリクス（ホーム） |
| 1 | 合意の回し方 | `consensus` | MHNG ラウンド再生（§4.4 Consensus Dynamics） |
| 2 | 共通図 | `shared` | *w* エクスプローラ（用語集・分類・IDレジストリ・重大度） |
| 3 | 3Dアンカー | `anchor` | *w* の一レンダリング（Viewer3D 流用） |
| 4 | 焼きなまし | `collapse` | collapse operator・不可逆デッドライン（§5） |
| 5 | 答え合わせ | `error` | 誤差信号・相関ガバナンス（§4.5） |

**命名規則（重要）:** `SYNAPSE_LoopArchitecture.md` §9 の dual naming に従い、**UI第一表示は必ず平易な日本語名**（観測／解釈／共通図／合意の回し方／答え合わせ）。CPC用語（o/z/w、MHNG、annealing）はサブラベル・ツールチップ・情報パネルの補足として添える。デモ相手にCPC語彙の前提知識を要求しない。

### 4.2 シナリオ時刻コントロール（全タブ共通）

ヘッダー直下に常設する**週スライダー**（W1〜W16）＋再生/一時停止/1週送りボタン。

- 全タブの表示は選択中の週 `week` の状態スナップショットを反映する（synapse01 の ANCHORS タイムラインの拡張版）
- 再生中は1週≒2秒で自動送り。タブを切り替えても時刻は保持
- `week` state はルートが持ち、全タブに props で渡す

## 5. タブ別仕様

### 5.0 タブ0「ループ」（ホーム）

**目的:** ループの全景を一枚で示す。デモの起点かつ健全性の常設監視画面。

- **中央:** ループ図（SVGで描画）。5ノード（観測→解釈→合意の回し方→共通図→答え合わせ→観測に戻る循環矢印）。選択週にアクティビティがあるノードをパルス強調。各ノードクリックで対応タブへ遷移
- **ノードのサブラベル:** 例「観測 *o* — 各ドメインの測定・試験・実車確認」
- **右側パネル:** 健全性メトリクス7指標（DataModel の `HEALTH` 参照）をカード表示。各カードは現在値＋スパークライン（W1〜選択週）＋緑/黄/赤判定
- **下部:** 選択週のアクティビティフィード（その週のラウンド・collapse・観測イベントを時系列リスト表示。クリックで該当タブへ）
- **左上の一言ピッチ:** §9 の一行ピッチを常時表示（「SYNAPSEは車両の共通図を正直に保つ：全チームが提案し、提案は実データで検証され、予測は現物と答え合わせされる」）

### 5.1 タブ1「合意の回し方」（MHNG ラウンド再生）

**目的:** 提案-受容プロトコルを具体的に見せる。デモの山場。

- **上部:** プロトコル種別フィルタ（日次統合／節目レビュー／フェーズ遷移交渉／外部認証）
- **中央（ラウンドビューア）:** 選択週のラウンドを1件ずつステップ再生
  - 話者ドメイン → 提案内容（用語定義・分類など）→ 聴者ドメイン の3カラム表示
  - 受容確率をバーで表示（シナリオ記録値）。「この提案は聴者自身の観測をどれだけよく説明するか」という説明文を添える
  - 受容/棄却の結果をアニメーション（受容→ *w* 成分の modes が更新される様子を右パネルに反映）
- **右パネル:** 対象 *w* 成分の現在状態（多峰分布を候補値の積み上げバーで表示、合意強度、staleness）
- **下部:** ドメインペア別の受容率分布ヒストグラム。1.0/0.0 に張り付いたペアには「儀式化の兆候」警告バッジ（健全性指標#1のインライン表示）
- **収束チャート:** 用語ごとの安定度曲線（KL風の模擬系列。W1〜選択週）

### 5.2 タブ2「共通図」（*w* エクスプローラ）

**目的:** 「共有されているのは3Dモデルではなく *w* である」を見せる。

サブビュー切替（4種）:

1. **用語集（living glossary）:** 係争語20語のカードリスト。各カード＝現在の定義候補（多峰なら複数併記＋支持ドメイン）、合意強度スコア、staleness ゲージ、最終物理接地週
2. **部品分類:** ドア系サブシステム24部品の分類表（新規/流用/一部変更）。流用部品は **collapse certificate** ドロワー（凍結時の設計荷重・規制基準・相関予測・サプライヤ工程状態の4項目）を開ける。証明書のない凍結部品には **fossil バッジ**（赤）＋fossil インベントリへの集約ビュー
3. **IDレジストリ:** persistent ID ↔ フェーズ別ジオメトリの対応表（コンセプト→クラスA→量産CADの3列。W12のフェーズ遷移交渉で対応が更新される様子が週送りで見える）
4. **リスク重大度:** 重大度付き *w* 成分のリスト。各行に「較正根拠＝提案ドメインの過去予測的中率」を表示

各 *w* 成分カード共通の表示要素: 多峰分布（積み上げバー）／合意強度／staleness（半減期ゲージ、物理接地でリセット）／温度／デッドライン週。

### 5.3 タブ3「3Dアンカー」

**目的:** synapse01 の3Dビューを「*w* の一レンダリング」として再文脈化する。

- **流用:** synapse01 `Viewer3D.jsx` の車両構築コード（BoxGeometry/CylinderGeometry 手組み、`meshes[partId]` 配列、useRef 保持、Raycaster 選択、マウス/ホイール操作）をそのまま流用。分解図・ワイヤー表示も残してよい
- **着色モード切替（4種、synapse02 の新規価値）:**
  1. 分類 — 新規/流用/一部変更/fossil で色分け
  2. 合意強度 — *w* 成分の consensus 値のグラデーション
  3. staleness — 鮮度（物理接地からの経過）のグラデーション。減衰した部位が退色していく表現
  4. 温度 — 焼きなまし温度（高温=赤、collapse済=青）
- **タイムライン:** 共通の週スライダーに連動して着色が変化（synapse01 の ANCHOR_PCTS パターンを週次スナップショットに置換）
- **部位クリック:** 右パネルに該当部位に紐づく *w* 成分（分類・証明書・関連ラウンド履歴）を表示
- **UIコピー:** ビュー上部に「この3Dモデルは共通図 *w* のひとつの描画です。アンカーの実体は永続ID＋合意状態です」と常時表示（本アーキテクチャの中心命題をUIで明言する）

### 5.4 タブ4「焼きなまし」（Collapse Timeline）

**目的:** 「ゲート＝一斉凍結」から「成分別焼きなまし」への転換を見せる。

- **メイン:** ガントチャート風タイムライン。行＝デッドラインを持つ *w* 成分、横軸＝W1〜W16
  - 各行に温度カーブ（デッドラインに向けて冷却）をヒートマップ帯で表示
  - collapse イベント（凍結確定）を ❄ マーカー、re-heat（一部変更による再加熱）を 🔥 マーカーで表示
  - デッドライン超過でまだ多峰の成分は**赤帯＋「collapse遅延」警告**（健全性指標#4）
- **凡例パネル:** 「ゲートとは宣言された成分集合の合意分布を収束させる焼きなましイベントである」の説明と、従来ゲート（全成分一斉）との対比図（静的SVG）
- **イベントクリック:** 該当成分の詳細（タブ2の該当カードへ遷移）

### 5.5 タブ5「答え合わせ」（誤差信号・相関ガバナンス）

**目的:** 「予測は必ず現物と突き合わせる。誤差が次の提案を駆動する」を見せる。

- **上部:** 予測-実測ペアの散布/系列チャート（航続距離・NVH の2領域。シナリオ中の物理試験イベント W6/W13 で実測点が入る）
- **トレースチェーン表示（重要）:** 乖離1件を選ぶと「物理観測 → 乖離登録 → 生成された提案 → *w* 更新」の連鎖を横並びのステップカードで表示。**「誤差は *o* を書き換えない。新しい提案を生む」**の注記を常設（append-only 原則の可視化）
- **下部:** ドメイン別予測的中率テーブル（この値がタブ1の受容重みの根拠になっている、という接続を説明文で明示）＋リスクトレースマトリクス（乖離×関連 *w* 成分×緩和策の表）

## 6. デザイン原則

- **配色・トーン:** synapse01 の `C` トークンを基調に踏襲（ダークテーマ）。synapse02 追加トークン（温度・staleness・ドメイン色）は DataModel §5 に定義
- **状態色の二重定義:** CSS用16進とThree.js用0x数値を必ず両方定義し同期させる（synapse01 の既知の規約）
- **共通部品:** synapse01 `Shared.jsx`（Bar/Pill/StatusPill/Card/CardTitle/Logo）を流用し、synapse02 で必要な `Gauge`（staleness半減期）/ `ModeBar`（多峰分布積み上げ）/ `TempStripe`（温度帯）を同ファイル様式で追加
- **説明の埋め込み:** 各タブに「この画面は何を見せているか」1〜2文のリード文を常設。デモポータルなので**画面自体が自己説明的**であること

## 7. 受け入れ基準（実装完了の定義）

1. `cd synapse02 && npm install && npm run dev` で起動し、6タブすべてが動作する
2. `npm run build` がエラーなく完了する
3. 週スライダー/再生で W1→W16 を通し再生でき、全タブの表示が週に追随する
4. タブ1で任意の週のラウンドをステップ再生でき、受容/棄却が *w* 成分表示に反映される
5. タブ2で fossil バッジ付き部品の collapse certificate 欠落状態と、流用部品の証明書4項目が確認できる
6. タブ3で4つの着色モードが切り替わり、部位クリックで *w* 成分パネルが開く
7. タブ4で collapse/re-heat イベントと「collapse遅延」警告が表示される
8. タブ5で乖離→提案→*w* 更新のトレースチェーンが1件以上たどれる
9. タブ0の健全性7指標が週に応じて変化する
10. すべてのUI第一表示が平易な日本語名である（CPC用語は補足表示に留まる）
11. シナリオの物語（DataModel §6）が通し再生で破綻なく読み取れる
