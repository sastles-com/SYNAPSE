## 1. プロジェクト概要

### 目的

新規車両開発における設計・実験・デザイン（スタイリング）の3領域の進捗を1つのポータルで統合可視化する。過去車種のデータをデータベースとして引用しながら、進捗の更新に応じて3D車両モデルもリアルタイムで変化する仕組みを構築する。

### コアコンセプト

> 3Dモデルは「可視化UI」ではなく、異なるドメインが同一のアンカーを介して収束する **情報統合点** であり、**議論のブリッジ** である。

### 対象ユーザー

| ユーザー | 主な用途 |
|---|---|
| 開発エンジニア（設計・実験） | 部位別進捗確認・課題登録・ドキュメント参照 |
| デザイナー（スタイリング） | CMF進捗・レンダリング履歴・デザインレビュー |
| プロジェクトマネージャー | 全体進捗・マイルストーン・課題管理 |
| 経営・役員層 | KPIサマリー・フェーズゲート承認・リスク把握 |

## 2. 本質的価値命題

### アンカーの定義

```
アンカー = 3Dモデルの特定部位 × 特定時点のスナップショット
```

アンカーは単なるファイルではなく、「ある時点で全ドメインが合意した成果物の状態」を指す。アンカーが変わるたびにデジタルスレッドに新しいノードが追加される。

### 3層の価値構造

**第一層 — アンカーとしての3Dモデル**  
3Dモデルの特定部位 × 特定時点の組み合わせが情報統合点となる。設計・実験・デザイン・経営の4ドメインが異なる言語で同じアンカーを参照することで、「どの版に基づいた議論か」という曖昧さが消える。

**第二層 — ドメイン間ブリッジ**  
- 設計が「このアンカーで寸法確定」と宣言
- 実験が「この形状で衝突試験を実施」と参照
- デザインが「このシルエットで最終スタイリング承認」と参照
- 経営が「このゲートで投資判断」と参照

**第三層 — デジタルスレッドとしての時間軸**  
過去プロジェクトの同部位のアンカー履歴が参照可能なため、「前回このブラケットで強度問題が出た」という知見が現在のアンカーに文脈として紐づく。これが「データベースとして引用する」という要件の本質。

## 4. 設計原則（3原則）

### 原則① デジタルスレッド（航空宇宙 PLM 型）

**参照元**: Dassault 3DEXPERIENCE / PTC Windchill のRFLP構造

**概念**: フェーズをまたいだデータの連続性確保。構想時のCADデータが実験結果・デザインレビューと紐付くアーキテクチャ。

| 層 | SYNAPSEへの実装 |
|---|---|
| UI層 | タイムラインスライダー — 過去アンカーを3D上で再現 |
| 統合層 | バージョンリンカー — CAD ver ↔ 試験結果 ↔ DR記録を自動紐付け |
| データ層 | スナップショットDB — `{partId, timestamp, domain, value}` のレコード積み重ね |

**フロントエンド実装ヒント**  
現在の3Dビューワーにタイムスライダーを追加。スライダーを動かすと各部位の色と進捗値がそのアンカー時点のスナップショットに切り替わる。

---

### 原則② ゲートウェイ構造（医薬品 PLM 型）

**参照元**: Siemens Teamcenter X / Aras PLM のフェーズゲート管理

**概念**: フェーズ間の明確なGo/No-Goゲート管理。SYNAPSEのマイルストーンをフェーズゲートとして可視化するUI拡張。

| 層 | SYNAPSEへの実装 |
|---|---|
| UI層 | フェーズゲートUI — 充足条件バー・承認ボタン・差戻しフロー |
| 統合層 | ゲートチェッカー — 設計/実験/デザイン3領域の充足率を自動判定 |
| データ層 | ゲート定義テーブル — フェーズ × 部位 × 条件・閾値・承認ログ |

**フェーズゲートの充足条件例（DR3）**

```
DR3 通過条件:
  設計進捗   ≥ 80%（全部位平均）
  実験項目   ≥ 60%（基本性能試験完了）
  デザイン   CMF承認済み
  課題       重大課題 = 0件
```

---

### 原則③ 予測リスク可視化（ACC / Construction IQ 型）

**参照元**: Autodesk Construction Cloud の Construction IQ

**概念**: 現状進捗だけでなく、遅延リスクの予測フラグをダッシュボードに表示。SYNAPSEのISSUESデータを活用した予測アラートに発展可能。

| 層 | SYNAPSEへの実装 |
|---|---|
| UI層 | 予測アラートパネル — 3Dモデル上にリスク着色（Current / Predicted の2レイヤー） |
| 統合層 | リスクスコアラー — 遅延パターンを過去比較して予測係数を算出 |
| データ層 | ISSUESデータ拡張 — `openedAt` + `resolvedAt` + 平均解決日数 → 予測ロジック |

**リスクスコアリングロジック（将来実装）**

```javascript
// 予測遅延リスクスコア（簡易版）
const riskScore = (part) => {
  const avgResolutionDays = part.pastIssues.map(i => i.resolvedAt - i.openedAt).average();
  const currentDelayDays  = today - part.plannedDate;
  const openIssueWeight   = part.openIssues.filter(i => i.severity === 'high').length * 1.5;
  return (currentDelayDays / avgResolutionDays) + openIssueWeight;
};
```

## 7. フロントエンド技術スタック

| レイヤー | 採用技術 | 選定理由 |
|---|---|---|
| 3Dビューワー | **Three.js** | Web標準・glTF対応・事例豊富 |
| UIフレームワーク | **React** | コンポーネント管理・パネル切り替えに適合 |
| 3Dモデル形式 | **glTF / GLB** | Web標準・Three.jsネイティブ対応 |
| 状態管理 | **Zustand**（推奨） | 軽量・3D選択状態の管理に適合 |
| スタイリング | **Tailwind CSS** | 迅速なプロトタイピング |
| CAD変換 | **glTF変換パイプライン** | CADライセンス不要のWeb配信 |

### セットアップ

```bash
npm install three
# Zustand・Tailwindは任意
npm install zustand
npm install tailwindcss
```

```jsx
import SynapsePortal from './SynapsePortal';

function App() {
  return <SynapsePortal />;
```

## 8. ユーザー別ビュー設計

### 開発エンジニア向け
- 3Dビューワーで担当部位を直接クリック
- サブポータルで設計・実験の詳細データにアクセス
- 課題登録・コメント投稿をポータル上で完結

### デザイナー向け
- 3Dビューワーでスタイリング対象部位を選択
- デザインサブポータルでスケッチ・レンダリング・CMFステータスを管理
- エクステリア/インテリア単位でのドリルダウン

### PM向け
- ダッシュボードで全体進捗・課題数・マイルストーンを一覧
- フェーズゲートUIで通過条件の充足率を確認
- 変更通知でリアルタイムに状況把握

### 経営・役員層向け
- KPIサマリーカード（全体進捗 / フェーズ / 課題数）
- 3Dモデルのヒートマップで「赤い部位が多いか少ないか」を直感的に把握
- フェーズゲートの最終承認アクション

## 10. データ定数（現在のモックデータ）

`SynapsePortal.jsx` 上部の定数を実データに差し替えることでバックエンド連携が可能。

### PARTS（部位データ）

```javascript
const PARTS = [
  {
    id: "body",
    name: "ボディ外装",
    pct: 72,          // 総合進捗率
    status: "progress", // done / progress / delay / issue / todo
    design: 75,       // 設計進捗 %
    test: 40,         // 実験進捗 %
    dsgn: 80,         // デザイン進捗 %
    owner: "田中",
    issue: null       // 課題テキスト or null
  },
  // ...
];
```

### MILESTONES（マイルストーン）

```javascript
const MILESTONES = [
  { label: "コンセプト承認", date: "2026/01/15", status: "done" },
  { label: "詳細設計完了",   date: "2026/07/31", status: "active" },
  // status: done / active / todo / target
];
```

### ISSUES（課題データ）

```javascript
const ISSUES = [
  {
    title: "バッテリー熱管理 発熱異常",
    owner: "鈴木",
    area: "駆動系",
    id: "#I-042",
    sev: "high",      // high / mid / low
    // 将来拡張: openedAt, resolvedAt, partId
  },
];
```
