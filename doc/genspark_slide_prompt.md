# Genspark AIスライド用プロンプト

> 使い方：以下をすべてコピーして、Genspark の「AI Slides（スライド生成）」にそのまま貼り付けてください。
> 1行目の「想定読者」と「枚数」だけ、必要に応じて書き換えると全体のトーンと粒度が変わります。

---

あなたはプロのプレゼンテーション作成アシスタントです。以下の【設定】【デザイン指針】【スライド構成】に従って、日本語のスライドデッキを作成してください。各スライドは「1枚＝1メッセージ」を徹底し、本文は箇条書き中心で簡潔に。事実関係は以下に記載した内容の範囲で正確に扱い、推測で固有名詞や数字を足さないでください。

## 【設定】
- 想定読者：社内のチーム内・技術／戦略共有（必要に応じて「経営・上長向け」または「自分用の整理」に変更可）
- 枚数：14枚程度
- 言語：日本語
- 目的：車両開発ポータル「SYNAPSE」の構想にあたり、「他業界に Backstage のような単一の真実源／統合ポータルの実例があるか」を横断調査した結果と、SYNAPSE への含意を共有する

## 【デザイン指針】
- スタイル：ミニマル・フラット。余白を広く、装飾は控えめ。図表中心。
- 配色（意味づけ）：青＝設計、緑＝実験、紫＝デザイン（SYNAPSE のドメイン色に合わせる）。ステータス表現は green/blue/amber/red/gray。
- 強調色は1〜2色に絞る。1スライドに色を使いすぎない。
- 図を積極的に使う：2軸のポジショニングマップ、レイヤー構造図、比較表、ハブ&スポーク図など。
- タイトルは短く、すべて文中は「sentence case」相当の自然な日本語表記。誇張表現（!や「革新的」等）は避ける。

## 【スライド構成】

### 1. タイトル
- タイトル：他業界に学ぶ「単一の真実源」— SYNAPSE 構想のための横断調査
- サブ：建築・創薬・航空・プラント・半導体に Backstage 相当はあるか

### 2. 導入：背景と問い
- SYNAPSE＝車両開発（NV-2026 EV セダン）の統合可視化ポータル構想。設計・実験・デザインの3ドメインを一枚に束ね、クリッカブルな3Dモデル＋ヒートマップで全ステークホルダーの単一の窓を目指す。
- 問題：専門のサイロ化により、ドメイン横断の不整合が後工程まで露見しない。
- 本資料の問い：同じ仕組みは他業界にも存在するのか。あるなら何を学べるか。

### 3. 分析の枠組み① — アンカーと境界オブジェクト
- アンカー＝成果物の構造（車両なら3D形状）を、情報の入口かつ状態表示面として使うもの。
- アンカーが機能する条件：その「部位分解」が組織の「責任分担」と一致していること（部位＝担当チーム）。
- これが、異なる専門集団が同じ面を指差して協働できる「境界オブジェクト（boundary object）」になる。価値の本質は可視化ではなく、サイロ間の翻訳コスト削減と意思決定の前倒し。
- 推奨ビジュアル：中央に「アンカー」、周囲に設計・実験・デザイン・経営の4ドメインが各自の見方で同じアンカーを参照するハブ&スポーク図。

### 4. 分析の枠組み② — 価値の2軸
- SYNAPSE型の価値は直交する2軸に分解できる。
  - 軸A：オブジェクト・アンカー価値（3Dモデル＋ヒートマップで、非専門家でも状態が見える）
  - 軸B：ドメイン横断統合価値（single source of truth）
- 物理的で分解可能な成果物がある業界（車両・航空・建築・半導体）は両軸が高い。製薬・ソフトのように物理オブジェクトが弱い業界は軸Bは高いが軸Aのアンカーを別途作る必要がある。
- 推奨ビジュアル：横軸＝軸A、縦軸＝軸Bの2×2散布図に各業界を配置。

### 5. 分析の枠組み③ — デジタルツインの成熟度
- アンカーの「時間性／ガバナンス」には段階がある。
  - デジタルモデル：手動更新の静的な仮想モデル（現状の SYNAPSE モックはここ）
  - デジタルシャドウ：実物→仮想に自動で一方向に同期
  - デジタルツイン：双方向。仮想側の操作・判断が実物に作用する
- 実データを入れた瞬間に「誰のデータか・何を赤とするか」の恒常的再交渉が始まる＝ガバナンスが要点。
- 推奨ビジュアル：3段の階段図（モデル→シャドウ→ツイン）。

### 6. 横断調査の全体像（比較表）
- 表形式で5業界を一覧化。列＝業界／Backstage相当の代表例／アンカーの正体／性格。
  - 建築：CDE（Autodesk Construction Cloud、Procore、Bentley ProjectWise、Oracle Aconex、ISO 19650準拠）／BIMモデル＋文書／フェーズ別スイートの集合
  - 創薬：Veeva Vault（eTMF・RIM・QMS・LIMS）／規制文書・記録＋知識グラフ／物理アンカー不在の文書中心バックボーン
  - 航空：PLM（Siemens Teamcenter、Dassault 3DEXPERIENCE、PTC Windchill）／3DモデルとEBOM／最も成熟したエンジニアリング・アンカー
  - プラント：AVEVA AIM＋CONNECT、Cognite Data Fusion、AspenTech、Palantir Foundry／P&ID・3D・タグ＋オントロジー／産業DataOps
  - 半導体：EDA（Synopsys、Cadence、Siemens EDA）、DDM（ENOVIA Synchronicity）／IPブロック階層・チップレイアウト＋PDK／ツインの連鎖

### 7. 航空 — 最も近い親戚（PLM＋デジタルスレッド）
- PLM が「単一の真実源たる製品データ・バックボーン」を担う。設計→システムズエンジニアリング→EBOM→変更管理→リリースを一本のデジタルスレッドに。
- EBOM はリリースゲートでベースライン化＝フェーズゲート。SYNAPSE の構造とほぼ同型。
- Boeing・Airbus・Lockheed など OEM は自前のデジタルスレッドも構築。

### 8. 建築 — CDE（ISO 19650）
- CDE＝中央集権的な情報リポジトリ＝単一の真実源。BIMモデルがアンカー。
- 重要な発見：どの単一プラットフォームも建造物の全ライフサイクルにわたる真実源にはなり切れず、ベンダーはフェーズ特化スイート群を作る方向。
- デジタルツインは Bentley iTwin／Autodesk Tandem 等。

### 9. 創薬 — Veeva Vault（文書中心のバックボーン）
- Veeva Vault が規制対象の文書・データの「デジタルバックボーン」＝単一の真実源。臨床（eTMF）・規制（RIM）・品質（QMS）・LIMS を一つに。
- 物理オブジェクトがないため、アンカーは文書・記録と知識グラフ（Neo4j等）。
- Vault間連携（RIM↔QMS↔LIMS）が課題＝後述の「縫合」。

### 10. プラント＆半導体 — オントロジー／ツインの連鎖
- プラント：AVEVA が P&ID・3D・タグを一つの資産情報ハブに。Cognite Data Fusion・AspenTech・Palantir Foundry が産業 DataOps として単一の真実源を構築。
- 半導体：EDA（Synopsys／Cadence／Siemens EDA）の上に設計データ管理（ENOVIA Synchronicity＝設計の単一バージョン）。設計→fab→パッケージ→テストの「相互接続ツインの連鎖」。PDK が設計とfabの接点。

### 11. Backstage 詳解 — ソフトウェア版 SYNAPSE
- Spotify が自社のマイクロサービス管理のため作り、2020年公開、CNCF へ寄贈。採用3,000社超・推定200万人。
- 中核＝ソフトウェアカタログ。エンティティ：Component／API／Resource／System／Domain／Group／User。関係：dependsOn、providesApi/consumesApi、ownedBy。
- 決定的な仕組み＝所有関係。全エンティティが一意の owner（Group/User）を持ち、「誰に聞くか」「アラート」「オンコール」を駆動。＝部位分解＝責任分担。
- 他に：テンプレート（Scaffolder＝新規サービスを数分で）、TechDocs、プラグイン、スコアカード（Soundcheck／Tech Insights）。
- 弱点：カタログはメタデータ（YAML）の質次第。腐敗するとポータルが死ぬ＝検証とスコアカードで運用が必要。

### 12. GitHub の位置づけ — アンカーではなく供給源
- GitHub の最小単位はリポジトリで、サービス／システムではない。リポジトリ分解は責任構造を写し取れず、コード一ドメインに閉じる。＝アンカーになれない。
- ただし下層の中心的な供給源：Backstage は catalog-info.yaml を自動走査し、GitHub Teams を所有者源に同期、Actions の状態を吸い上げる。
- 車両に喩えると、GitHub＝設計ドメインのCAD＋変更レビューの源。3Dアンカーに当たるのは Backstage のカタログ。
- 推奨ビジュアル：上に Backstage（体験・アンカー層）、下に GitHub／CI・CD／可観測性／インシデント／組織図が供給源として並び、矢印が下から上（一方向＝シャドウ）。

### 13. 二つの発見＋ 最も Backstage 的な実例
- 発見1：「複数アンカーの縫合」は実在する製品カテゴリ。建築には複数CDEを同期する CDE Sync、創薬には Vault 間連携、Cognite は「単一の巨大ツインをやめ、小さく目的特化したツインを多数」と明言。成熟した現場ほど縫合へ向かう。
- 発見2：「生きたツイン」と「静的シャドウ」の区別は業界自身の言葉。Palantir は書き戻し（write-back）可能で双方向なものだけが本物のツイン、可視化だけはシャドウと切り分ける。AVEVA も常時同期する生きた表現と定義。
- 精神的に Backstage に最も近いのは Palantir Foundry の Ontology（部品・機械・SKU・患者等を第一級オブジェクト化＋双方向）。ただし Backstage はオープン／ボトムアップ／所有グラフ型、物理産業はほぼ専有ベンダースイートという違い。

### 14. SYNAPSE への含意と提言
- 車両 SYNAPSE の真の親戚は Backstage ではなく、航空・自動車の PLM＋デジタルスレッド／ツイン。日産も既に何らかの PLM を運用しているはず。
- だから SYNAPSE はカタログを作り直すのではなく、PLM・実験・スタイリングのソース群の上に乗る「横断的な境界オブジェクト体験層」として立つ（Backstage↔GitHub と同じ関係）。
- 白地：PLM はエンジニア向けで重く、経営・デザイナー・横断ステークホルダー向けの「単一の窓」は苦手。そこが SYNAPSE の領域。
- 次の一手：(1) ガバナンスを先に固める（状態データに出自・所有者・鮮度のメタを付与＝Backstage の owner＋検証＋スコアカード相当）。(2) PLM の上に乗せる参照アーキテクチャ、または Palantir 型オントロジーの車両版オブジェクトモデルを試設計。
- 推奨ビジュアル：横軸＝トポロジー（単一↔複数縫合）、縦軸＝ガバナンス（静的↔生きた）の2×2に「現状（静的・単一）」から「次の一手（生きた・単一）」へ上向きの矢印。
