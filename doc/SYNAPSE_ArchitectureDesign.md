# SYNAPSE アーキテクチャ設計ドキュメント
## Vehicle Development Intelligence Portal — 設計原則と統合構造

**文書の目的：** SYNAPSEが実現しようとする価値の構造的定義と、それを支える4層アーキテクチャ原則を記述する。フロントエンドプロトタイプから将来のバックエンド統合へ向けた設計指針として機能する。

---

## 1. SYNAPSEが束ねる「2つの独立した価値」

SYNAPSEを抽象化すると、性質の異なる2つの価値が重なっているシステムとして整理できる。

### (A) オブジェクト・アンカー型ナビゲーション

成果物そのもの（クリッカブルな3Dモデル＋ヒートマップ）を、情報の入口かつ状態表示面として使う発想。

- **本質：** 「専門家でない人でも、モノを見れば状態が分かる」という認知的民主化
- **対象ユーザー：** エンジニア・デザイナー・経営層という認知レベルの異なる人々を同一画面で統合
- **現在の実装：** Three.js 3D Viewer（クリッカブルパーツ＋ヒートマップ）

### (B) ドメイン横断の状態統合（Single Source of Truth）

設計・実験・デザインという、本来サイロ化した専門領域の進捗を一枚に束ねる情報統合の価値。

- **本質：** 各ドメインのDBと連携し、開発全体の状態を単一の真実として提供
- **対象：** 設計（設計部門）・実験（実験部門）・デザイン（デザイン部門）の3ドメイン統合
- **現在の実装：** PARTS / MILESTONES / ISSUES の定数データ（将来DB接続予定）

### この区別が重要な理由

他分野での類似システムとの距離感は、(A)(B)のどちらをどれだけ必要とするかで決まる。**両方を高水準で必要とする分野は実はそれほど多くない。**

```
                    (B) ドメイン横断統合 の必要性
                         低                    高
                    ┌──────────────────┬──────────────────┐
               高   │  建設 / BIM      │  ★ SYNAPSE       │
  (A) オブジェクト  │  3Dは強い        │  航空宇宙 PLM     │
     アンカーの     │  施工1ドメイン中心│  両方を高水準で  │
     必要性         ├──────────────────┼──────────────────┤
               低   │  半導体 EDA      │  医薬品 / 食品    │
                    │  3Dアンカー不要  │  統合は強いが     │
                    │  抽象的フロー管理│  3Dアンカー弱     │
                    └──────────────────┴──────────────────┘
```

**SYNAPSEの構造的条件：** 「成果物が物理オブジェクトとして明確に存在し、かつ複数の専門ドメインが並走する」開発プロセスにおいてのみ、(A)(B)の両立が意味を持つ。これは同時に、SYNAPSEの構造が転用可能な領域の境界線でもある。

## 2. 4層アーキテクチャ原則

4つの機能原則は、独立した機能ではなく**階層的に積み上がる統合アーキテクチャ**として理解する必要がある。

```
┌─────────────────────────────────────────────────────────────────┐
│  ④ 予測リスク可視化                          【出力層】          │
│                                                                  │
│  ①②③を統合することで初めて成立する。                          │
│  統合設計作業の連携を促す「議論の種」を見える化する。            │
│                                                                  │
│  参照事例: Autodesk Construction IQ / BIM × ML リスク検出        │
├─────────────────────────────────────────────────────────────────┤
│  ③ ゲートウェイ（デザインレビュー相当）      【判定層】          │
│                                                                  │
│  各ドメインの相互作用を検証し、                                  │
│  Go/No-Go の判定根拠を構造化する。                               │
│                                                                  │
│  参照事例: FDA-TRACK CDER Dashboard / 医薬品フェーズゲート       │
├─────────────────────────────────────────────────────────────────┤
│  ② デジタルスレッド                          【接続層】          │
│                                                                  │
│  各ドメインDBと連携し、データの流れ・トレーサビリティ・           │
│  滞りを検出・管理する。                                          │
│                                                                  │
│  参照事例: Siemens Teamcenter / 航空宇宙PLMデジタルスレッド      │
├─────────────────────────────────────────────────────────────────┤
│  ① デジタルツイン                            【アンカー層】       │
│                                                                  │
│  開発者・デザイナー・経営層が視覚的・感覚的に合流できる          │
│  共通の基準点。物理とデジタルを繋ぐ基盤。                        │
│                                                                  │
│  参照事例: McKinsey Digital Twin Framework / Siemens 製造DT      │
└─────────────────────────────────────────────────────────────────┘
```

## 3. 各層の詳細定義

### ① デジタルツイン — アンカー層

**役割：** 異なる専門領域の開発者・デザイナー・経営層が視覚的・感覚的に合流できる**共通の基準点（アンカー）**として機能する。

**機能：**
- 3DモデルはCADデータそのものではなく、「現物の状態」と「開発の進捗」を同時に体現するインターフェース
- 現物・実験結果・設計変更をリアルタイムで反映することでデジタルとリアルを繋ぐ
- 専門知識を持たないステークホルダーでも、モデルを見るだけで開発状態を把握できる認知的民主化を実現

**SYNAPSEにおける現在の実装：**
- Three.js による3D Viewerプロトタイプ（静的可視化）
- パーツクリッカブル機能
- ヒートマップカラーリング（Blue=設計 / Green=実験 / Purple=デザイン）

**将来の発展形：**
- 各パーツのCADデータ・実験計測データ・センサーデータとの接続
- NV-2026の「リビングモデル（Living Model）」化

---

### ② デジタルスレッド — 接続層

**役割：** フェーズをまたいでデータの連続性・トレーサビリティを確保する**データフロー管理機構**。

**機能：**
- 各ドメインのDB（設計CAD / 実験計測システム / デザインレビュー記録）を接続
- 変更の伝播・データの滞り・断絶箇所を検出・管理
- トレーサビリティの確保：「この設計変更が、どの実験データに基づき、どのデザイン判断につながったか」を追跡可能にする

**デジタルツインとの関係：**
- デジタルツインが「空間的アンカー」であるのに対し、デジタルスレッドは「時間的・データ的接続」を担う
- 両者は相補的：デジタルツインが「今どこにいるか」を示し、デジタルスレッドが「どこから来てどこへ向かうか」を示す

**SYNAPSEにおける現在の実装：**
- PARTS・MILESTONES・ISSUESデータ定数（フロントエンドで分離された構造）
- 将来のバックエンドDB統合のためのデータスキーマの萌芽

**将来の発展形：**
- 設計CAD系DB ↔ 実験計測系DB ↔ デザインレビュー系DBのリアルタイム接続
- 変更影響の自動伝播アラート

---

### ③ ゲートウェイ — 判定層

**役割：** 各フェーズの境界においてドメイン間の相互作用を検証し、**Go/No-Goの判定根拠を構造化する**機能。

**機能：**
- デザインレビューに相当するフェーズゲート管理
- 「設計変更が実験結果に与えた影響」「デザイン要件と実験データの整合性」といったドメイン横断の依存関係を明示的に評価
- 判定根拠をデジタルスレッドを通じて自動集約

**デジタルスレッドとの関係：**
- デジタルスレッドが収集したデータをゲートウェイが評価・判定する
- ゲートウェイの判定履歴がデジタルスレッドに蓄積され、将来の予測リスク可視化の根拠となる

**フェーズゲート構造（医薬品開発との対比）：**

```
医薬品: 前臨床 → Ph.I → Ph.II → Ph.III → FDA承認
            ↑        ↑              ↑           ↑
          Go/No-Go  Go/No-Go     Go/No-Go    Go/No-Go

SYNAPSE: 設計構想 → 基本設計完了 → 実験完了 → デザイン承認 → 量産判定
              ↑            ↑              ↑              ↑
            Go/No-Go    Go/No-Go      Go/No-Go       Go/No-Go
```

**SYNAPSEにおける現在の実装：**
- MILESTONESデータ（フェーズゲートの萌芽）

**将来の発展形：**
- 各ゲートへのドメイン間整合条件の紐付け
  - 例：「実験合格率 ≥ 95%」「デザイン承認ステータス = APPROVED」「設計変更影響スコア ≤ 閾値」
- Go/No-Go 判定UIと経営層向けレポート自動生成

---

### ④ 予測リスク可視化 — 出力層

**役割：** ①②③の統合によって初めて成立する**出力層**。統合設計作業の連携を促す「議論の種」を見える化する。

**機能：**
- 現状の進捗表示だけでなく、将来リスクを先行フラグアップ
- デジタルツイン（現状の空間的把握）＋ デジタルスレッド（過去の変化パターン）＋ ゲートウェイ（判定履歴）を統合して出力
- 「どのパーツが・どのドメイン間の連携において・どのタイミングでリスクを持つか」を可視化

**3層からの入力：**

| 入力層 | 提供する情報 | 予測への寄与 |
|---|---|---|
| ① デジタルツイン | 現物の状態・実験反映 | 現状の空間的把握 |
| ② デジタルスレッド | データフロー・変化パターン | 過去の遅延・断絶パターン |
| ③ ゲートウェイ | ドメイン間相互作用・判定履歴 | 類似ゲートでの失敗パターン |

**SYNAPSEにおける現在の実装：**
- ISSUESデータ（予測リスク可視化の萌芽）

**将来の発展形：**
- リスクスコアを3D Viewerのヒートマップに重ねて表示
- 類似パーツの過去遅延パターンに基づく先行アラート
- 「このパーツの実験完了が遅れると、デザインレビュー日程に N日影響する」という因果連鎖の可視化

## 4. 現在のSYNAPSEと将来アーキテクチャの対応表

| 機能層 | 現在の実装（フロントエンドプロトタイプ） | 将来の実装（バックエンド統合） |
|---|---|---|
| ① デジタルツイン（アンカー） | Three.js 3D Viewer + ヒートマップ（静的） | CAD/実験データとリアルタイム同期するリビングモデル |
| ② デジタルスレッド（接続） | PARTS / MILESTONES / ISSUES 定数データ | 設計・実験・デザインDB間のリアルタイム連携 |
| ③ ゲートウェイ（判定） | MILESTONESの進捗表示 | ドメイン間整合条件付きGo/No-Goゲート |
| ④ 予測リスク可視化（出力） | ISSUESリスト表示 | ヒートマップ重畳 + 先行アラート + 因果連鎖可視化 |
